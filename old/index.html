Perl5 Alien

History

Alien technology in Perl is all about dependency management.

Perl modules that have requirements declare their dependencies in the distribution metadata. Then when you install something using a CPAN client like cpanm, all of its dependencies get installed first automatically. But what do you do if your dependency is not on CPAN? This most frequently becomes an issue with C libraries used by XS modules.

But it could also be java byte code used by Inline::Java or a rust library used by FFI::Platypus. There is no way to specify a non-CPAN or "alien" dependency in the distribution metadata.

The state of the art for this used to be documentation. A comment in the POD that you need to install libfoo-dev in Debian and foo-devel in RedHat, and well lets be honest, you're screwed if you are on Windows.

=head1 INSTALL

On Debian install libfoo-dev on RedHat install foo-devel.  On Windows,
sorry sucker.

Also if you don't have root or a package manager meh.

=cut

And this is if you are lucky, many authors don't even bother with instructions for non-CPAN dependencies.

In 2003 Artur Bergman proposed something better. His idea was to use the Alien:: namespace for external, non-CPAN dependencies. Others have picked up this idea and run with it.

In 2012 Joel Berger created Alien::Base, which acts as a base class that Alien:: modules can use to do the heavy lifting.
